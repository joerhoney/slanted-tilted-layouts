Slanted & Tilted Layouts

In this repo we explore the possibilities and challanges of creating a unique sort of layout, using tilted (rotated) and slanted (skewed) boxes.

Using Tailwind

As we don't want to lose time reinventing the wheel and we do want to exercise best practices, let's use a utility-first CSS library to do the heavy lifting for us. I rejected Tailwind for several years, thinking it was too similar to inline styles. But Tailwind at least addresses one major deal-killer with inline styles, which is no query support. What good is a layout today that isn't responsive?

The initial approach here was to introduce supplementary classes which would cover the few things Tailwind can't really do. But as I gradiently learned more about Tailwind, the amount of custom CSS needed kept dwindling, and I ended up deleting over half of it. But this made me apprciate what Tailwind does even more and I learned a LOT more about Sass and CSS as well (including why some CSS preprocessors are not recommended with Tailwind). So I don't regret having gone to such great lengths only to undo a bunch of it.

IMPORTANT: Please note that this is a work in progress and could be better optimized. New interations of this experiment will aim to utilize Tailwind as recommended by its documentation.

Slanted Boxes

Let's start with the easier one. "Slanted" boxes are pretty straight forward. Using Tailwind we need the transform class to do any slanting (or any box shape manipulation). It doesn't do anything by itself, it just sets the stage, so to speak. Then we can use any of Tailwind's transform classes we want. (The way they made that work using CSS variables is pretty ingenius if you ask me.)

transform skew-y-12

For this demo I chose skew-y-12, which skews the element by 12 degrees along the Y axis (up/down). I actually need 14 for my project, but that gets into customizing Tailwind, which is beyond the scope of this demo. So 12 degrees will do, and we just have to remember that number when we add our Tailwind classes.

Nesting and Inheritance

Transforms behave like they are inherited, in a sense, even though technically they're not. When you apply any transform to an element (skew, rotate, scale, etc.), all content within is also transformed in that way. So in order to skew a box and not it's content we have to "unskew" that content. So we need the opposite of a 12 degree skew, which would be -12 degrees. With Tailwind, all we have to do to convert a value to negative is add a hyphen (-) in font of it.

transform -skew-y-12

If we then want to "reskew" something nested with that, we simply use the first set of classes again.

  • transform skew-y-12 = "slant" or "reslant"
  • transform -skew-y-12 = "unslant"

Tilted Boxes

You can immediatly see the difference "tilted" boxes have from "slanted" when you introduce columns (which you'll only see here on widers screens). It's almost such a different look that you could arguably say don't go together, design-wise. It's obvious with anything else box shaped as well, such as images. But certainly it's a really cool look on its own and quite popular, as can be observed in much of today's marketing. I'm a fan of it myself, so I wanted to do something to make its many caveats easier to cope with.

transform rotate-12

Just like the "slanted" box, all we need is Tailwinds transform classes, but this time we use rotation. And again, to "unrotate" what's inside we just add that hyphen in front of the rotate-12 class

transform -rotate-12

If we do want to combine slanting and rotating, and we want a consistent angle throughout the page, the skew and rotate values match, so we're sticking with 12 degrees for tilting as well, here.

Tilt Caveats

Because tilted boxes are actually rotated, we now get into some problematic geometry, some of which CSS provides little to no access to resolve mathmatically. Rotating a tall block of text for instance means one verticle end (top or bottom) swings off in one direction and the alternate end swings off the other way, naturally.

To compound the matter, text blocks are very frequently much taller and narrower on mobile devices than on devices with wider screens. It seems rotation is also harder for the system to calculate than skewing is.

For these reasons, I would recommend reserving "tilting" for wider devices, as much as possible. This is one of the first areas where Tailwind really shines, with its media query syntax.

Media Queries

With Tailwind, simply prepend md: to the class you only want applied to the medium breakpoint on up, or use another one of its breakpoint specifier (sm:, lg:, xl:, 2xl:). Example:

skew-y-12 md:skew-y-0 md:rotate-12

The above classes will skew 12 degrees (on the Y axis), then "unskew" and rotate 12 degrees at the medium breakpoint. "Unskewing" is necessary as both skew and rotate can be applied concurrently. Neither one overrides the other. In this way, we are following a mobile-first principle, as skew-y-12 is set for all devices, and the rest is ignored by the smaller devices. Smaller devices would be more prone to exhibiting a lag when having to process overrides, so we let larger, more powerful devices deal with that.

I've designed most of these clases to follow the same pattern and use these same breakpoint prefixes.

Box Corners

We can (and should) avoid making our tilted boxes too tall, or break their inner content into short boxes, individually rotated. However, even if we do this, we will have protruding box corners with anything we rotate (because everything in HTML is box shaped, even if you can't see it). To deal with this, the best solution I have found is to cut the corners off with a wrapper, using overflow-hidden (a Tailwind class).

  • Keep boxes from growing too tall, or...
  • Break it down: individaully rotate shorter items within.
  • Use media queries to avoid tilting on smaller screen sizes.
  • Trim corners with a wrapper that hides overflowing.

Utilitarian Flexibility

Aside from the reasons mentioned above, the utilitarian approach we're using in this demo gives us much flexibility.

I wanted to be able to do things like reverse row directions and insert an number of columns.

You might have noticed this row is reversed (except on small devices), yet we still haven't had to come up with any costume CSS for it.

So far Tailwind covers everything we have aimed to achieve, and we can control and understand it right within our HTML.

So if we can do almost everything we need to in this demo with Tailwind CSS, what's left and what's the point of this demonstration?

Focusing again on the corners of a skewed or rotated box, have you noticed we are actually now dealing with a triangular space?

And if you change the width of this viewport, have you noticed those triangular spaces scale with a proportional aspect-ratio?

Triangular Padding

Tailwind's default measurement system mostly uses rem units. This is fine in most cases, but not very practical for maintaining our necessary aspect-ratio. You could customize Tailwind to use something like vw or cqw, but this isn't recommended. Generally, Tailwind should be used as-is and only added to, if anything. Tailwind allows us to use arbitrary values in our HTML such as pt-[12cqw] (padding-top: 12cqw;), and that may be considered better practice. I'm not a fan of the idea, myself. To adhere to an aspect-ratio, the container width needs to be observed so the proper space can be calculated accordingly. We really need to use the cqw unit or at least vw. Without this, the tops and bottoms of the tilted or slanted elements will either be too big on some devices, or the text will spill out on other devices, or both.

So our first custom class will address the triangular space we get from skewing and rotating. Now to make it easy-ish to remember these new classes, and since most classes used in the project are those of Tailwind, let's follow Tailwind's syntax as much as we can. Tailwind syntax is short, so for "triangular space" let's just use tri-. Then, as this is a type of padding, let's borrow Tailwind's pt, pb, etc. And since this is custom, we don't need to support tons of variants like tri-pt-6, tri-pt-7, ..., etc. Let's just use the container aspect-ratio that will work for our project, which in this case is 12cqw. (I just played around with it to find that number, but usually this would be close to the number of degrees you are using for your slants and tilts.)

Here are the classes for the job:

  • tri-pt = padding-top: var(--triangle-pos);
  • tri-pb = padding-bottom: var(--triangle-pos);
  • tri-py = padding-top: var(--triangle-pos); padding-bottom: var(--triangle-pos);

var(--triangle-pos) is a CSS variable I set, which equals 12cqw. This should generate the right amount of padding for the triangular space, based upon the width of the container element.

Containers and Query Units

Note that this (the above, regarding cqw) means the container has to be a container. Yep, it's a thing, and it's not just any old wrapper. A container is explicitly declared as such. Otherwise, how would container query units (cqh, cqw and cqi) know which "container" to base itself on when it's nested multiple levels deep. If it were to always based itself on the size of its immediate parent (which is what I thought it did), that would be useless in some situations, or could get very messy as we would tend to declare too many containers. Fortunately, the viewport essentially acts as though it is automatically declared as a container, meaning if you have no containers defined, cqw equates to vw, cqh equates to vh, etc.

Columns

Here we've created a column class with container-type: inline-size, which gives us something to reference for the size of the triangular padding, rather than referring to the viewport size. I wasn't able to find a Tailwind class to set the container-type (maybe because it's pretty new), hence I created my own. Because of this, I made my column class serve several other purposes.

  • I added a couple other column-related CSS rules to it as well, to reduce the amount of classes needed when declaring a column in my HTML.
  • With its unmistakable name, it's much easier to find my columns in my HTML and see its bounaries, which is nice.
  • Because I've added the bare minimum CSS to make it a column, I pretty much of have another useful utility now, with nearly as much flexibility as Tailwind's classes.

Background Images

Background colors are straightforward, but when adding images to these boxes we need to understand how they are affected by the transforms. For instance, if you add a CSS background image to a tilted (rotated) box, the image will be tilted along with it. This may be just what we want in some cases, which is great! If not, the image will need to be placed on (or in) a box that is "untilted" (rotated in the reverse direction, so as to straighten it out again).

Now when we have a straightened or untilted image inside a tilted box we often get the corner problem again. That is, the image corners want to spill out. In some cases we can fix this by adding overflow-hidden, to the tilted box. But depending on your edge case, there may still be another "corner" problem (or more of a "hole" problem).

In probably most cases, we want the image to fill the container, but an untilted image inside a tilted box leaves holes. And this is the case even with all the CSS we normally use to make an image fill an area. CSS doesn't see what we see. If you tell it to rotate something, it just rotates it in place. It sees no consequential need to resize anything if we didn't ask it to. So to force the image to occupy those holes, we have to tell the browser to scale it up larger than the tilted box it sits within.

Adding a dedicated element for it is often the best way to generating a background in these boxes. We just need to treat it as its own layer and ensure that it sits beneath the forground content. Scaling a background image can be done in several ways, but the most challenging aspects of it are balancing "enough" scaling vs "too much", and for any variety of screen sizes, without stretching the image.

I'll spare the details of how I derived the CSS for it, but I created a class for that:

tri-cover

It works in most cases, but it's far from perfect. There's some compromise (usually a little more upscaling than necessary) and sometimes the image gets stretched (mostly only those with portrait orientation). I might play around with this class more at some point, because I feel like there has to be some way to base it on the container's height as well as the width.

Backgrounds in Columns

For the most part, we can get the background to fill a column with positioning, hiding overflow and sometimes upscaling. But in a tilted row, the first and last columns tend to have holes (or gaps, I suppose). Rotating the box moves some of the corners outward (as we have covered above), but what about the opposing corners that are brought inward? This doesn't harm scrolling, but it unsightly, isn't it?

So I created classes for that too:

  • reach
  • reach-left
  • reach-left
  • reach-0
  • reach-right-0
  • reach-right-0

The *-0 classes are for overriding the others at higher breakpoints. So md:reach-left won't start doing it's thing until the medium breakpoint, then if you needed to deactivate that at the next breakpoint you could use lg:reach-left-0.

The way these work is by simply stretching the side edges out beyond their boundaries (by 50% of the container width) and then adding that much padding on that edge. The background, if properly set up, ignores the padding and bleeds off the page like with the box, while the inner content is remains constrained by the box within the padding.

Variables

I love CSS variables. You can of course modify this css for your project if you choose to use it, but I used variables a lot to automatically calculate certain spacing and positioning properties. This makes it very easy to modify all of this for other projects, or at least if you need to change the angle. So if you want to use Tailwind's skew-y-6 (instead of skew-y-12), you can likely change the variables to --angle-pos: 6deg, --angle-neg: -6deg, --triangle-pos: 6cqw, and you should have all the right triangle padding, triangle background "cover" (fill), etc. (Or you shouldn't have to adjust much from there).

To follow the patterns of Tailwind more closely, these classes could have been written with numeric variants, like tri-py-3, instead of basing it all on these variables. Perhaps there is a good way to programatically do that in a tailwind.config file. But here on Codepen, supporting the same numeric increments as Tailwind would require a great deal of additional SCSS, and that is definitely not the point of this demonstration.
